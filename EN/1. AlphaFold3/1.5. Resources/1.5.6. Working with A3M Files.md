# Working with A3M Files

[[Home|Home]] > Resources
🇺🇦 [[UA/1. AlphaFold3/1.5. Ресурси/1.5.6. Робота з A3M файлами|Українська]]

> **A3M** (`alignment format`) is a compact `multiple sequence alignment` (MSA) format typically produced by `HHblits`, `HHsuite` tools, or `MMseqs2`/`ColabFold` pipelines. Unlike plain `FASTA`, it stores both aligned columns and insertions relative to the `query` by using mixed case.

## What A3M is and why it is useful

`A3M` is useful when a pipeline needs not just a list of homologs, but an alignment explicitly indexed to the `query` sequence.

In practice it is used for three things:

- storing MSAs compactly without expanding every insertion into every row;
- separating `aligned columns` from insertions that do not correspond to `query` token positions;
- building `AlphaFold`-style features such as `msa`, `has_deletion`, `deletion_value`, and `profile`.

This matters in AF2/AF3-like pipelines because the model does not consume a human-readable alignment. It consumes tensors where every position must map cleanly to the `query` length.

## How to read A3M syntax

| Element | Meaning in A3M | What to do during featurization |
| --- | --- | --- |
| Uppercase letters `A-Z` | aligned residues that correspond to `query` columns | keep as tokens |
| Lowercase letters `a-z` | insertions in the homolog relative to the `query` | do not make separate tokens; accumulate as insertion count |
| `-` | gap in an aligned column | encode as a gap / deletion token |
| `.` | optional placeholder in insert-only columns in some tools | usually ignore |
| `>` | start of a new record | begin a new MSA row |

The key idea is that the raw string length of an `A3M` row does not have to equal the `query` length, because lowercase letters are not separate `query positions`.

## Short example

```text
>query
MKTAYIAKQRQISFVKSHFSRQDILD
>hit_exact
MKTAYIAKQRQISFVKSHFSRQDILD
>hit_with_insertion
MKTAYIAKQRQISFVKSHFasRQDILD
>hit_with_gap
MKTA-IAKQRQISFVKSHFSRQDILD
```

Here:

- `hit_exact` matches the query exactly;
- in `hit_with_insertion`, the characters `as` are an **insertion** between two aligned columns;
- in `hit_with_gap`, the `-` symbol marks a missing residue in one aligned column.

After decoding to the `query` axis:

- lowercase letters are removed from the token row;
- `-` is kept as a gap;
- the number of removed lowercase residues is carried into the `deletion_value` feature for the next aligned position.

## How A3M differs from FASTA and Stockholm

| Format | What it stores | Typical use case |
| --- | --- | --- |
| `FASTA` | standalone sequences without alignment semantics | raw protein or nucleotide sequences |
| `A3M` | compact MSA with mixed-case insertions | HHblits, HHsearch, ColabFold, AF2/AF3 preprocessing |
| `Stockholm` | MSA plus column and family annotations | Pfam/Rfam, curated alignments |

So `A3M` is mainly an **engineering format for models and search pipelines**, whereas `Stockholm` is better when richer annotations are needed.

## Practical parsing in Python

The easiest way to break an `A3M` pipeline is to call `upper()` on the whole file. That destroys the insertion signal.

```python
def parse_a3m(path: str) -> list[tuple[str, str]]:
    """
    Reads an A3M file and preserves mixed case.
    Returns [(header, raw_sequence), ...].
    """
    records = []
    header = None
    buf = []

    with open(path, "r") as fh:
        for line in fh:
            line = line.rstrip()
            if not line:
                continue
            if line.startswith(">"):
                if header is not None:
                    records.append((header, "".join(buf)))
                header = line[1:]
                buf = []
            else:
                buf.append(line)

    if header is not None:
        records.append((header, "".join(buf)))
    return records


def decode_a3m_row(raw_seq: str) -> tuple[str, list[int]]:
    """
    Converts one raw A3M row to:
      1. aligned sequence on the query axis;
      2. insertion counts before each aligned position.
    """
    aligned = []
    insertions = []
    pending = 0

    for ch in raw_seq:
        if ch.islower():
            pending += 1
        elif ch == ".":
            continue
        else:
            aligned.append(ch)
            insertions.append(pending)
            pending = 0

    return "".join(aligned), insertions


records = parse_a3m("query.a3m")
query_header, query_raw = records[0]

for header, raw_seq in records[:3]:
    aligned, ins = decode_a3m_row(raw_seq)
    print(header)
    print("aligned:", aligned)
    print("insertions_before_cols:", ins[:10])
```

This mirrors the core logic of AF3-style preprocessing:

- `aligned` becomes the row on the `query_len` axis;
- `-` remains an explicit token;
- `insertions` can be transformed with `log1p` into a `deletion_value` feature.

## Common A3M operations

```bash
# Stockholm -> A3M
reformat.pl sto a3m query.sto query.a3m

# A3M -> alignment FASTA without Stockholm annotations
reformat.pl a3m fas query.a3m query.aln.fasta

# Filter redundant rows by sequence identity
hhfilter -i query.a3m -o query_id90.a3m -id 90 -cov 75

# Build HHM profile from A3M
hhmake -i query.a3m -o query.hhm
```

In practice:

- `reformat.pl` is the convenient converter between MSA formats;
- `hhfilter` reduces redundancy before feature construction;
- `hhmake` turns `A3M` into a profile `HMM` for `HHsearch`-compatible steps.

## Where A3M files come from in practice

An `.a3m` file is usually not downloaded as a standalone dataset. It is typically **generated by searching large sequence databases** or returned by a service that already ran that search for you.

| Source | How the A3M is obtained | When it is useful |
| --- | --- | --- |
| `UniRef90` | `HHblits` search over high-identity clusters | fast start, close homologs |
| `UniRef30` | second `HHblits` pass over broader clusters | more distant homologs, wider coverage |
| `UniProt` | `Jackhmmer` → `Stockholm` → `reformat.pl sto a3m` | maximum evolutionary depth |
| `ColabFold MMseqs2 server` | service returns a ready-made `A3M` | when you do not want to install hundreds of GB of local databases |
| curated family alignments | existing `Stockholm`/`FASTA` alignment converted to `A3M` | manual or domain-specific MSA pipelines |

Practically:

- `UniRef90` and `UniRef30` provide most of the homolog coverage for AF-style pipelines;
- `UniProt` is useful when rare or weaker homologs matter;
- `ColabFold` is the easiest "sequence -> ready `.a3m`" option;
- curated alignments are appropriate when you already have a family MSA and only need to convert it into an engineering-friendly format.

### Minimal practical source set

If the goal is simply to get a `.a3m` for one protein, one of these two paths is usually enough:

1. `ColabFold MMseqs2 server` — the easiest option without local infrastructure.
2. `HHblits` + `UniRef90`/`UniRef30` — if you need a local, reproducible pipeline.

If maximum coverage matters:

1. `HHblits` on `UniRef90`;
2. `HHblits` on `UniRef30`;
3. `Jackhmmer` on `UniProt`;
4. merge and filter the resulting `.a3m` files.

## How A3M is used in AlphaFold 3

In `AF3`, `MSA` plays a smaller role than in `AF2`, but `A3M` is still important as the intermediate format between sequence search and tensor features.

The practical chain is:

1. `HHblits`, `Jackhmmer`, or `MMseqs2` find homologs.
2. The search tool returns `A3M`, or a format that is easily converted into `A3M`.
3. The first row (`query`) defines the `query axis`.
4. Lowercase characters are removed from the token representation, but their meaning is retained.
5. The pipeline computes `msa`, `has_deletion`, `deletion_value`, and `profile`.
6. Those features go into the `input embedder` and reduced `MSA module`.

For AF3-like implementations, this means `A3M` is not the final human-facing format. It is the **bridge between evolutionary search and numerical featurization**.

## Limitations and common mistakes

- Do not treat `A3M` as ordinary `FASTA`; mixed case is meaningful.
- Do not blindly strip `-` if you need positions aligned to the `query`.
- Not all viewers render uppercase vs lowercase equally clearly.
- `A3M` is mainly convenient for protein MSAs; richly annotated family alignments are often better handled in `Stockholm`.
- If the `query` is not the first row, many downstream scripts become incorrect or ambiguous.
